## 上下文

当前工具已经具备 `sync/check/doctor` 三个命令和声明式配置解析能力，但配置创建仍依赖手写 JSON。首次接入时，用户需要自己理解 `source.kind`、`targets`、`mode` 字段，容易出现路径和字段错误；同时编辑器缺少统一 schema 来源，导致提示和校验体验不一致。

本次设计在不改变 v1 核心同步语义（single-source、`link/copy`、`check`、`dry-run/force`）的前提下，补齐“配置初始化 + 编辑期校验”能力。

## 目标 / 非目标

**目标：**
- 新增 `init` 交互式命令，指导用户生成可直接执行的配置文件。
- 提供常用工具预置选择，一键生成 `targets` 默认配置。
- 提供并维护官方 JSON Schema，支持在配置文件中通过 `$schema` 引用。
- 保持 `source.kind=canonical|tool`、`link/copy`、`check` 既有语义不变。
- 测试策略继续采用 `rstest` + TDD（先写失败用例再实现）。

**非目标：**
- 不引入云端模板市场、远程 registry 或在线模板拉取。
- 不引入 GUI 配置向导。
- 不改变同步引擎的执行模型（Planner/Executor）与既有退出码约定。

## 决策

### 决策 1: 新增 `init` 子命令并采用成熟第三方交互库
- 选择：在 CLI 中新增 `context-sync init`，基于成熟交互库实现问答流程；业务层仅依赖抽象的 Prompt Adapter 接口（收集 source 类型、源路径、工具选择、mode、输出文件路径、覆盖确认）。
- 理由：减少与业务无关的终端交互样板代码，提升可维护性和一致性；通过 Adapter 可在测试中注入假实现，保持可测试性。
- 备选：使用 Node 原生 `readline/promises` 手写交互。缺点是流程控制和输入校验细节分散在业务代码中，维护成本更高。

### 决策 2: 复用现有 source 模型，显式保留 `source.kind=canonical|tool`
- 选择：`init` 只负责生成符合现有模型的配置，不新增 source 类型。
- 理由：保持与现有 `parseConfig` 契约一致，避免破坏 `sync/check`。
- 备选：在 `init` 阶段引入新的抽象 source 类型。缺点是会扩大迁移面。

### 决策 3: 预置工具映射采用可维护的静态 registry
- 选择：在代码中维护常用工具预置（如 codex/claude/cursor/copilot/continue/cline）到默认路径模板的映射，`init` 允许多选并生成 `targets`。
- 理由：满足“一键配置”的核心诉求，并与现有 adapter 体系保持一致。
- 备选：完全自由输入目标配置。缺点是首用门槛高、易出错。

### 决策 4: JSON Schema 以仓库文件为单一真源，并发布到 GitHub 可访问地址
- 选择：在仓库维护 `context-sync.schema.json`（版本受 Git 管理），文档提供 GitHub Raw 地址作为 `$schema` 推荐引用。
- 理由：编辑器可直接消费，且 schema 与代码可同 PR 演进。
- 备选：仅在 npm 包内发布 schema。缺点是编辑器引用和跨仓共享不便。

### 决策 5: `init` 生成配置默认包含 `$schema`，但解析层保持向后兼容
- 选择：生成的新配置写入 `$schema` 字段；加载配置时允许该字段存在且不影响旧配置解析。
- 理由：新用户立即获得补全/校验体验，旧用户零破坏升级。
- 备选：不写 `$schema`。缺点是无法兑现“帮助用户编辑 JSON 配置”的目标。

### 决策 6: 保持 link/copy/check 语义不变，仅新增“由 init 生成配置”的覆盖测试
- 选择：`init` 不改变引擎语义，`check` 仍需覆盖 link 的 symlink mismatch 与 copy 的 modified/missing/extra。
- 理由：将变更范围限制在配置创建与校验体验，降低回归风险。
- 备选：同时重构 check 输出模型。缺点是风险与测试成本显著上升。

### 决策 7: 测试采用 rstest + TDD，分层验证
- 选择：先新增失败测试，再实现；覆盖参数解析、交互流程、生成配置快照、schema 校验与兼容性。
- 理由：CLI 交互和文件输出路径分支多，测试先行可约束行为并避免回归。
- 备选：实现后补测。缺点是行为约束不稳定。

## 风险 / 权衡

- [交互流程过长导致放弃初始化] → 提供默认值与预置模板，问题最小化。
- [第三方交互库升级导致行为变化] → 固定主版本并增加 CLI 快照/流程测试，升级时走回归验证。
- [GitHub Raw 地址变更或引用不稳定] → 文档中同时给出版本化地址（tag）与最新地址（main），并建议生产环境使用版本化 URL。
- [预置路径不适配个别团队目录结构] → 允许用户在向导中覆盖路径，且保留手写配置能力。
- [新增 `$schema` 与严格校验工具存在兼容差异] → 解析层忽略未知字段，仅校验业务必需字段。

## 迁移计划

1. 新增 schema 文件与 schema 发布地址约定（README + 示例配置）。
2. 引入第三方交互库并新增 `init` 命令参数解析与交互式向导模块（通过 Prompt Adapter 隔离库实现）。
3. 新增配置写入器（含 `$schema` 注入、覆盖确认、路径解析）。
4. 先编写 `rstest` 失败用例：
   - init 交互输入 -> 生成期望配置；
   - 已有配置且未确认覆盖 -> 失败；
   - 生成配置可被 `sync/check` 正常读取；
   - link/copy check 失败场景保持既有行为。
5. 实现功能并使测试通过，补充文档与示例。

回滚策略：
- 若交互命令在特定环境不稳定，可临时隐藏 `init` 命令入口，同时保留手写配置与现有命令不受影响。
