## 新增需求

### 需求:提供开源 npm CLI 的核心用户文档
系统必须提供面向陌生用户的完整入门文档，至少覆盖项目价值、安装方式、快速开始、命令参考与配置示例。

#### 场景:新用户安装与首轮执行
- **当** 用户首次访问仓库并按照 README 执行安装与示例命令
- **那么** 用户必须能够在不查阅源码的情况下完成 `context-sync` 的一次有效执行

#### 场景:命令参考可定位
- **当** 用户需要查看 `sync`、`check`、`doctor` 以及 `dry-run`、`force` 参数行为
- **那么** 文档必须在单一入口或可追踪链接中提供准确说明

### 需求:文档必须明确 link/copy 差异与 check 失败条件
系统文档必须准确说明 `link` 与 `copy` 的行为差异，并明确 `check` 的典型失败条件。

#### 场景:link/copy 行为差异可理解
- **当** 用户阅读配置与行为说明
- **那么** 文档必须明确 `mode=link` 与 `mode=copy` 的文件系统效果差异

#### 场景:check 失败条件可预期
- **当** 用户使用 `check` 排障
- **那么** 文档必须明确至少包含 `symlink mismatch` 与 copy drift（`modified/missing/extra`）两类失败条件

### 需求:提供开源协作文档基线
仓库必须提供开源协作所需的最小文档集合，至少包含许可证、贡献指南、行为准则、安全策略与变更记录策略。

#### 场景:外部贡献者准备提交 PR
- **当** 外部贡献者准备参与贡献
- **那么** 仓库必须提供可执行的贡献流程与行为规范说明

### 需求:提供可验证的文档与元信息一致性检查
系统必须提供可执行的验证步骤，确保 README、命令说明和包元信息与当前 CLI 行为保持一致。

#### 场景:CI 验证文档与元信息
- **当** 维护者在 CI 或本地执行项目校验命令
- **那么** 流程必须包含对文档与 `package.json` 关键字段完整性的检查，并在不一致时返回失败
