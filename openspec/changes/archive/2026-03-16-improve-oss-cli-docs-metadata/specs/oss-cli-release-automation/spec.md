## 新增需求

### 需求:必须提供自动化 CI 发包流程
系统必须通过受控 CI 流水线执行 npm 发包，禁止将人工本地发包作为正式发布路径。

#### 场景:触发正式发包
- **当** 维护者在受支持的发布触发条件下执行发布（例如推送符合约定的 tag 或触发 release workflow）
- **那么** 流水线必须自动完成构建、测试与 npm publish，并记录可追溯的执行日志

#### 场景:发布前校验失败
- **当** 流水线中的构建或测试失败
- **那么** 发包步骤必须被阻断且发布状态标记为失败

### 需求:必须自动生成或更新 Release 与 Tag
系统必须在发布流程中自动创建或更新 Git Tag 与 GitHub Release，确保版本工件与说明可追踪。

#### 场景:新版本发布
- **当** 发布流程执行成功
- **那么** 系统必须存在与版本一致的 tag 与 release 记录，并包含该版本的核心变更说明

### 需求:必须提供构建来源证明
系统必须提供可验证的“由自动化流水线构建并发布”的证明机制，并在仓库公开展示。

#### 场景:用户验证来源证明
- **当** 用户在仓库主页查看开源证明信息
- **那么** 用户必须能够通过公开 badge 或链接验证该版本由自动化流水线构建发布

### 需求:README 必须维护核心数据 badges
系统必须在 README 中维护开源 npm 包的核心数据 badges，并保持其与实际状态一致。

#### 场景:README badges 完整
- **当** 用户打开 README
- **那么** 页面必须至少展示 CI 状态、npm 版本、npm 下载量、License 与来源证明 badge

#### 场景:badge 状态失真
- **当** badge 对应的数据源不可用或与当前仓库状态不一致
- **那么** 维护流程必须在 CI 校验阶段失败并提示修复
