## 上下文

当前项目已实现 `context-sync` 的核心能力（`source.kind=canonical|tool`、`link/copy`、`check`、`dry-run`、`force`），但对外发布面仍偏向内部开发者视角：
- `README.md` 信息结构不完整，缺少面向 npm 用户的安装、快速开始、命令手册、配置示例分层。
- `package.json` 仍为私有包形态（`private: true`，名称与描述不适合作为公开发布产物），缺少标准开源字段。
- 缺失关键社区协作文档（贡献、安全、行为准则、变更记录、许可证）。
- 缺失标准开源发布链路（自动发包、自动 Release/Tag）和构建来源可验证证明（provenance badge）。

目标是将项目提升为“可公开发布、可被陌生用户直接采用、可被社区协作维护”的开源 npm CLI 基线，同时不改变一期同步能力语义。

## 目标 / 非目标

**目标：**
- 建立开源 npm CLI 的文档最小完备集，并让 README 成为单一入口。
- 建立 npm 发布所需元信息最小完备集，确保包可发现、可追踪、可联系、可声明约束。
- 建立自动化发布链路，使发包与 Release/Tag 由 CI 流水线驱动并可审计。
- 建立开源证明与数据展示基线，使 README badges 可反映真实构建与发布状态。
- 将 `context-sync` 现有行为语义（尤其 `source.kind=canonical|tool`、`link/copy/check`）映射到公开文档并保持一致。
- 为后续发布自动化留出结构化入口（例如 changelog、release 说明基础）。

**非目标：**
- 不修改 `sync/check/doctor` 的核心执行逻辑与输出协议。
- 不引入云端模板拉取与远程 registry（保持一期非目标不变）。
- 不在本次变更中新增 CLI 子命令或配置字段。

## 决策

1. 文档体系采用“根文档 + 专题文档”双层结构。
- 选择：README 负责价值说明、安装、最短路径示例和跳转索引；细节命令与开发协作放入独立文档。
- 备选：全部信息堆叠在 README。
- 取舍：双层结构降低首屏复杂度，便于长期维护与贡献。

2. 包元信息按 npm 开源发布最小字段集强约束。
- 选择：补齐 `name/description/license/repository/homepage/bugs/keywords/engines/publishConfig/files/bin/exports`。
- 备选：仅补齐最少能发布的字段。
- 取舍：一次性补齐减少后续重复返工和发布歧义，提升检索与信任度。

3. 发布动作采用“CI 驱动 + 受保护凭据 + 可追溯产物”。
- 选择：通过 GitHub Actions 执行构建、测试、发包与 release 流程，禁止手工本地发包作为正式流程。
- 备选：保留人工发包为主，CI 仅做校验。
- 取舍：CI 驱动可显著降低人为失误与权限外泄风险，并支持 provenance 证明。

4. Release/Tag 自动化与 changelog 生成解耦实现。
- 选择：先保证 tag/release 自动创建与内容注入（可由约定模板或工具生成），再逐步增强语义化发布策略。
- 备选：一次性引入完整 release 管理平台与复杂语义版本自动推导。
- 取舍：分步落地降低引入成本，先满足“可自动发版、可追踪”硬要求。

5. 现有同步语义不变，仅补齐“文档可发现性与声明一致性”要求。
- 选择：在 `context-strategy-sync` 增量规范中新增文档一致性需求，不改已有行为需求。
- 备选：将文档要求单独放在新增 capability，完全不触碰现有 capability。
- 取舍：同步能力与对外说明天然耦合，补充一条“行为必须被文档准确描述”的要求更可审计。

6. 对 `source.kind=canonical|tool` 与 `link/copy/check` 的说明采用“统一术语 + 示例驱动”。
- 选择：所有文档中统一使用 capability 已定义术语，并提供最少一组 canonical/tool 与 link/copy 示例。
- 备选：在不同文档使用近义词描述。
- 取舍：统一术语可减少误解，直接支持后续测试与 issue 排查。

7. README badge 采用“最小有效集 + 自动更新”。
- 选择：至少包含 CI 状态、npm 版本、npm 下载量、License、构建来源证明（provenance）badge。
- 备选：仅保留静态描述，不放 badge。
- 取舍：badge 提升可见性与可信度，但必须依赖自动化更新避免失真。

## 风险 / 权衡

- [风险] 文档更新滞后于代码实现，导致用户误用。
  → 缓解措施：在规范中要求命令、配置与示例必须与当前 CLI 行为一致，并在 CI 中加入文档一致性检查（可后续落地为任务）。

- [风险] 元信息补齐后仍存在 npm 命名冲突或可用性问题。
  → 缓解措施：发布前执行 `npm view <name>` 可用性检查，必要时调整公开包名并同步文档。

- [风险] 补充开源协作文档提升维护负担。
  → 缓解措施：先建立最小模板与责任边界，避免过度流程化；仅保留高价值条目。

- [风险] 自动发包流程配置不当可能导致误发布。
  → 缓解措施：对发布分支/Tag 增加保护规则，使用环境审批与最小权限 Token，并在 workflow 中加入 dry-run 阶段。

- [风险] provenance/badge 数据源中断导致 README 误报。
  → 缓解措施：badge 来源统一使用可稳定访问的数据源，并在 CI 增加链接与状态校验。

- [权衡] 本变更不提升核心功能，仅提升发布与协作可用性。
  → 价值在于降低外部采用门槛与后续发布风险，是功能增长前的必要地基。
